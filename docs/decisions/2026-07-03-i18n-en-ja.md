# 英語・日本語の2言語対応とし、バッファテキストは翻訳しない

- Date: 2026-07-03
- Type: ADR
- Status: Active
- Author: masahiro.kasatani

## Context

Vim ユーザーは世界中にいるため英語対応でリーチが大きく変わる一方、日本語の Vim コミュニティも厚い。ただし全てを翻訳対象にするとコストが膨らみ、問題コントリビュートの参入障壁（ADR `challenges-as-json-with-ci-verification` で下げたもの）を再び上げてしまう。翻訳の範囲と運用ルールを決める必要があった。

## Decision

- 対応言語は **en / ja**。en をベース言語とし、ja が欠けている箇所は en にフォールバックする
- **バッファテキスト（`initial.text` / `expected.text`）は翻訳しない**。言語に関係なく ASCII のコード風テキストに統一する
- 問題データの翻訳対象（`title` / `prompt` / `hints`）は `LocalizedText` 型（`en` 必須・`ja` 任意）としてファイル内に両言語を持つ。`ja` 欠落は CI エラーにせずレポート出力とし、翻訳はメンテナ側で補完する
- i18n ライブラリは導入せず、辞書オブジェクト + フォールバックを自作する。言語切り替えはブラウザ言語判定 + 手動トグル（localStorage 保存）。URL パスによるロケールルーティングはしない
- 解説文（オペレータ / モーションの説明）は core 側に言語別の説明辞書を持ち、ロケールは引数で受け取る
- README / CONTRIBUTING は `README.md`（英語）+ `README.ja.md`。CLAUDE.md は開発者向け内部ドキュメントとして日本語のまま翻訳しない

## Consequences

- 英語圏・日本語圏の両方にリーチできる
- バッファ実装は ASCII 前提のまま単純に保たれる
- コントリビュータは片方の言語だけで問題を投稿できる（CONTRIBUTING に明記）
- 翻訳欠落の補完はメンテナの継続タスクになる（欠落一覧レポートで管理）

## Alternatives Considered

- **バッファテキストも翻訳** — CJK 文字を扱うと `w` の単語境界判定・文字幅描画・カーソル計算が複雑化し、最小サブセット（ADR `minimal-vim-subset`）の前提が崩れるため却下。実務で Vim が編集するのはコードや英文であり、練習素材としても ASCII が自然
- **両言語を必須にする** — 「JSON を置いて CI が通れば OK」という参入障壁の低さが崩れるため却下
- **ロケールごとに問題ファイルを分ける** — 片方だけ修正されて内容がズレる事故が必ず起きるため却下
- **react-i18next 等の導入** — UI 文字列の数に対して過剰なため却下
