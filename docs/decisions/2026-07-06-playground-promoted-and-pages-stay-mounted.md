# プレイグラウンドを一般ユーザー向け機能に昇格し、両トップレベル画面を常時マウントする

- Date: 2026-07-06
- Type: ADR
- Status: Active
- Author: masahiro.kasatani

## Context

プレイグラウンドはこれまで開発用の素振りページ（Step 6 の名残）だった。これを「タイマーも判定もない素振り場」として一般ユーザー向けに昇格し、リザルト画面から `p` キーでその場のお題の初期テキストを持ち込んで試せるようにする。これには「プレイグラウンドから戻ったとき、元のリザルト画面（`r` でリトライ可能な状態）に復帰できる」という要件が伴う。

## Decision

- **`App.tsx` の「ゲーム / プレイグラウンド」タブは、両方を常時マウントしたまま `hidden` 属性で出し分ける。** 従来は非表示側のページを丸ごとアンマウントしていたため、プレイグラウンドに寄り道するとゲーム側の状態（進行中のラウンド、リザルト画面）が失われていた。両方を常時マウントすることで、この状態が自然に保持される
- **各画面は `active: boolean` プロパティを受け取り、自分の keydown リスナーをそれで止める。** 常時マウントするだけでは、非表示側の画面も `window` に張った keydown リスナーを持ち続けてしまい、同じキー入力が両方の画面で二重に処理される。`GamePage` → `MenuScreen` / `LevelRound` の `ChallengeRound` / `LevelSummaryScreen`、`PlaygroundPage` のそれぞれが `active` を受け取り、`false` のときはリスナー本体で早期 return する（`ChallengeRound` は ref 経由、他は素朴な条件分岐 — 頻度に応じた使い分けは既存の方針を踏襲）
- **`ChallengeRound` のタイマーも `active` でゲートする。** 非表示のままタイマーが動き続けると、プレイグラウンドに寄り道している間に気づかず時間切れになる。`active` が `false` の間は `setTimeout` を予約しないことで、視界に入っていない間は時間が進まないようにした
- **プレイグラウンドはタイマー・判定・Attempt 記録を一切持たない。** 打った操作は `recordAttempt` を一度も呼ばない。学習ループ（解説を読む → 試す → リトライ）の受け皿という位置づけを、データの面でも徹底する
- **プレイグラウンド内の UI 操作（テキスト差し替え・リセット・プリセット選択）はクリックとテキスト入力フィールドのみで行い、キー割り当てを一切しない。** これは CLAUDE.md の「キーボード第一」原則の意図的な例外であり、理由は単純: プレイグラウンドではキーボードの入力を残らず Vim エンジンに渡す必要があり、UI 操作用のキー（例えば "リセット" を `r` に割り当てる等）と Vim コマンドの名前空間が完全に衝突するため。テキスト入力フィールドにフォーカス中は `fieldFocusedRef`（state ではなく ref — フォーカス/ブラーは常にユーザー操作から同期的に起きるため、古い値を読む心配がない）でゲートし、Vim エンジン側のキー処理から隔離する
- **リザルト画面の `p` キー/キーキャップは、その問題の `initial.text`/`initial.cursor` を新しいオブジェクトとして渡すだけで、ラウンド自身の状態には一切触れない。** `PlaygroundPage` 側は「渡された seed オブジェクトの参照が変わったら適用する」という単純な `useEffect` で受け取る。同じ問題で `p` を2回押しても、毎回新しいオブジェクトが生成されるため正しく毎回適用される
- **プレイグラウンドの keydown ハンドラは `ChallengeRound` のような ref 全面駆使アーキテクチャを踏襲しない。** ラウンド／リザルトのフェーズ切り替えという競合の起きうる状態がプレイグラウンドには存在しないため、単純な「状態が変わるたびにリスナーを張り替える」設計で十分と判断した（差分を小さく保つ判断でもある）

## Consequences

- 画面をまたいだ状態保持のために、`GamePage` / `MenuScreen` / `LevelRound` / `LevelSummaryScreen` / `PlaygroundPage` のすべてが新たに `active` プロパティを受け取るようになり、シグネチャがやや膨らんだ。ただし個々の変更は「関数の先頭で `if (!active) return` を足す」程度に収まっている
- プレイグラウンドの「直近5件の実行履歴」「pending 状態表示」「打ったキー列のリアルタイム解説」は、いずれも既存の `explain` / `explainSequence`（`src/core/explain.ts`）をそのまま再利用しており、新しい解説機構は作っていない
- ASCII 判定は正規表現ではなく `charCodeAt` を1文字ずつ確認する素朴な関数にした（Biome の `noControlCharactersInRegex` に抵触するため）
- 「リセット」が戻す対象は固定の初期テキストではなく、直近にセットされたテキスト（プリセット・手動セット・リザルトからの持ち込みのいずれか）にした。プレイグラウンドは「今取り組んでいる題材」を試行錯誤する場であり、常に同じデフォルトに戻る必要はないため

## Alternatives Considered

- **ゲーム画面の状態（特にリザルト表示中のラウンド状態）を `GamePage` より上位（`App.tsx` やコンテキスト）に持ち上げ、アンマウント/リマウントを許容したまま復元する** — `ChallengeRound` のローカル state（buffer/keys/mistakes/commands/score/displayStreak/timeLeft 等）をすべてシリアライズ可能な形に持ち上げる必要があり、影響範囲がはるかに大きい。「常時マウント + active ゲーティング」の方が既存のコンポーネント構造に対する変更が小さく、動作の説明も単純だったためこちらを選んだ
- **プレイグラウンドの UI 操作にもキーを割り当てる（例: リセットを大文字の `R` にする等）** — Vim エンジンに全キーを渡すという素振り場の趣旨と衝突し、意図しないタイミングでリセットや差し替えが起きるリスクがあるため却下
